# C2KV 压缩 Tool Definition 

## 1. 数据集与任务构造

在 agent-llm-traces 上划分训练集和测试集，输入 user query + tool definitions，模型输出 function/tool call

- 按 session 划分 train / eval
- 每个 session 内 tool definition 相同，因此每个 session 最多采样若干 span
- 把工具定义作为 context，把用户输入作为 prompt，把模型的原始回复作为 target 
- target 使用模型原始回复，包含：
  - Thought / 自然语言推理
  - Action / `<tool_call>` 工具调用

过滤规则：

- `max_tool_definition_tokens=10000` 超过 10000 token 的 tool definition 暂时过滤
- `max_doc_length=1024`
- `max_doc_num=10`
- max input length  2048

## 2. 模型训练设置

基础模型：Qwen3-4B-Instruct-2507

训练设备：Ascend 910B3 8 卡

训练方式：只训练 C2KV / gist 相关参数，base model 冻结

训练速度：每 step 时间 ≈  34.5 秒

## 3. 对比方法

### 3.1 C2KV

训练后的 C2KV 压缩器。

流程：

```text
tool definition
-> 按 1024 token 切成多个 document chunk
-> generate_gist 生成压缩 KV
-> blend_gist_key_values 融合压缩 KV
-> 当前对话基于 past_key_values 生成 tool call
```

压缩率通过 `override_ratio` 控制：

| ratio | 含义                                               |
| ----: | -------------------------------------------------- |
|     2 | 约 2 个 tool definition token 压成 1 个 gist token |
|     4 | 约 4 个 token 压成 1 个 gist token                 |
|     8 | 约 8 个 token 压成 1 个 gist token                 |

### 3.2 C2KV Untrained

未训练的 C2KV baseline。

含义：

- 使用同一个 base model。
- 初始化 gist_q / gist_k / gist_v / gist_embed。
- 不加载训练后的 gist 权重。

作用：判断训练是否真的让压缩器学到了 tool definition 信息。

### 3.3 Full Baseline

无压缩 baseline。

做法：将完整 tool definition 放入 prompt / KV prefill。

作用：作为质量上限参考。

### 3.4 Truncate Baseline

同等倍率的截断 baseline。

做法：直接保留 tool definition 的前缀 token。

作用：判断 C2KV 是否优于简单截断。

## 5. 结果分析

### 5.1 训练后的 C2KV 明显优于未训练 C2KV

|     4 |       0.3208 |         0.2642 | +0.0566 |
|     8 |       0.3019 |         0.2547 | +0.0472 |

说明训练后的 gist 参数学到了 tool definition 的有效信息。

### 5.2 C2KV 优于截断 baseline

**工具名准确率**：

| ratio |   C2KV | Truncate |      结论 |
| ----: | -----: | -------: | --------: |

|     8 | 0.3019 |   0.3019 |      持平 |

**工具调用率**：

| ratio |   C2KV | Truncate |

|     2 | 0.5189 |   0.4340 |
|     4 | 0.5189 |   0.4340 |

说明 C2KV 比简单截断更容易让模型进入 tool-call 模式。

### 5.3 压缩率越高，质量整体下降

### 5.4 TTFT 降低

假设tool definition 可离线压缩，C2KV 在线请求时不需要再计算 `tool_compress_sec`，TTFT 主要来自 system prefill 和 blend。

| 方法     | ratio | TTFT (s) |
| -------- | :---- | :------- |
| c2kv     | 2     | 0.1581   |
| c2kv     | 4     | 0.1555   |

如果工具定义已提前离线提取，C2KV 的首 token 前 KV 准备时间明显低于 full 和 truncate。

## 6. Session-level 多轮评测

### 6.1 目的

单轮评测中，每个 span 都独立计算 tool definition，无法体现 Agent 多轮场景中的 KV cache 复用。实际推理中，同一个 session 内 tool definition 通常保持不变：

```text
同一个 session:
第一次 span 构建 system/tool KV
后续 span 复用这段 KV
每个 span 仍使用自己的完整 input.messages
```

### 6.2 设置

- 按 session会话 分组。
- 每个 span 的 `input.messages` 仍然使用数据集中记录的完整累计输入。
- 每个 session 只构建一次 system/tool KV，后续 span 默认复用该 KV cache。

评测规模：

```text
session 数：20
span 总数：367
```

### 6.3 指标

| 指标                     | 含义                                                     |
| ------------------------ | -------------------------------------------------------- |
| 会话级宏平均工具名准确率 | 先计算每个 session 内工具名准确率，再对 session 等权平均 |
| 会话成功率               | 一个 session 内所有 span 的 tool name 都正确的比例       |
| 平均首次错误轮次         | 每个 session 第一次出现工具名错误的轮次平均值            |

### **6.5 多轮结果分析**

#### 6.5.1 C2KV 在多轮场景中质量优势更明显

span-level 工具名准确率：

| **方法**    | **工具名准确率** |
| ----------- | :--------------- |
| c2kv 2x     | 0.4005           |
| c2kv 4x     | 0.3815           |
| c2kv 8x     | 0.3270           |
| full        | 0.3869           |
| truncate 2x | 0.2916           |
| truncate 4x | 0.2752           |
| truncate 8x | 0.2616           |

C2KV 2x / 4x 明显优于 truncate baseline，并接近 full baseline。

其中 C2KV 2x 的 span-level 工具名准确率略高于 full，但 session macro 指标中 full 仍然更高，说明 full 在不同 session 间更稳定。

#### 6.5.2 C2KV 更容易触发 tool call

**工具调用率：**

| **方法** | **ratio** | **工具调用率** |
| -------- | --------: | -------------: |
| c2kv     |         2 |         0.4659 |
| c2kv     |         4 |         0.4578 |
| full     |         1 |         0.4632 |
| truncate |         2 |         0.3324 |
| truncate |         4 |         0.3243 |

C2KV 的工具调用率接近 full，显著高于 truncate。说明 C2KV 不仅保留了工具名信息，也保留了“当前应该调用工具”的信号。

#### 6.5.3 默认复用下，KV 构建成本已经较低

默认 session-level 推理中，每个 session 只构建一次 system/tool KV，因此构建成本被摊到同一个 session 的多个 span 上。

会话构建耗时：

| **方法** | **ratio** | **会话构建耗时(s)** |
| -------- | --------: | ------------------: |
| c2kv     |         2 |              0.9593 |
| c2kv     |         4 |              0.7998 |
| c2kv     |         8 |              0.7311 |
| full     |         1 |              1.0493 |
| truncate |         2 |              0.4851 |
| truncate |         4 |              0.3816 |
| truncate |         8 |              0.3746 |

truncate 的构建时间最低，因为只保留部分前缀 token；C2KV 需要额外执行 `generate_gist` 和 `blend`，因此构建时间高于 truncate，但仍然低于 1 秒级别。full 需要预填充完整 tool definition，因此构建时间最高。

#### **6.5.4 端到端耗时主要由 generate 阶段主导**

单样本均摊总耗时：

| **方法** | **ratio** | **单样本生成耗时(s)** | **单样本均摊总耗时(s)** |
| -------- | --------: | --------------------: | ----------------------: |
| c2kv     |         2 |               11.4791 |                 11.5314 |
| c2kv     |         4 |               11.3012 |             **11.3448** |
| c2kv     |         8 |               12.4451 |                 12.4850 |
| full     |         1 |               11.6121 |                 11.6693 |

默认复用后，构建成本已经很小，整体耗时主要由 generate 阶段决定：

```text
单样本生成耗时约 11-13s
单样本均摊总耗时与 generate 耗时非常接近
```

C2KV 4x 的均摊总耗时最低，C2KV 2x 的质量最好且耗时接近 full。truncate 虽然构建最便宜，但生成阶段更慢，整体并没有获得明显速度优势。后续需要统计 `generated_tokens` 和 `tokens_per_second` 来进一步区分“生成更长”和“每 token 计算更慢”。

#### **6.5.5 会话成功率较低，因为多轮全对很难**

会话成功率要求一个 session 内所有 span 的 tool name 都正确。由于本次平均每个 session 约 18 个 span，该指标非常严格。

结果中：

```text
C2KV 会话成功率：0.00
full 会话成功率：0.05
truncate 会话成功率：0.00 - 0.05
```

平均首次错误轮次多数接近 1，说明很多 session 从第一轮就可能出错，而不是后续轮次才因为历史累积逐渐退化。

## 7.目前结论

训练后的 C2KV 能比较有效学习 tool definition 信息

在工具名准确率和工具调用率上优于简单截断 baseline

在多轮复用场景中，C2KV 2x/4x 能在接近 full baseline 质量的同时降低 KV 构建成本



下一步，可以选择性地使用c2kv进行压缩，提升精度。比如重要程度不同的工具用不同压缩率，或根据字段的重要程度压缩

针对不同类型的上下文，采用不同的压缩和复用策略

与 Memory 结合，将 agent 记忆按工具定义、任务轨迹、用户偏好、失败经验和反思 信息等类型进行分层压缩与管理
